# Releasing Nimlat

This is the maintainer procedure for publishing Nimlat source code and official desktop binaries. Nimlat is
source-available proprietary software, not open-source software. Only the copyright owner may publish official builds;
see [LICENSE](LICENSE) and [NOTICE.md](NOTICE.md).

The first release should be assembled as a GitHub draft by a maintainer-triggered GitHub Actions workflow, tested on
every target operating system, and published only after every gate below passes. The explicit manual commands remain the
fallback procedure and the behavioral contract for that automation; the workflow must not weaken any gate.

## Release model

- One app version maps to one immutable Git commit, one `v<version>` tag, and one GitHub Release in `Kunepro/nimlat`.
- A GitHub Release is one release with multiple downloadable assets, not one release per operating system. It contains
  the source snapshot generated by GitHub plus all unsigned macOS, Windows, and Linux artifacts.
- The same Windows app version may also be distributed through the Microsoft Store as an MSIX/AppX package built from
  the same commit. Microsoft signs the Store package after certification; that signature does not sign or otherwise
  alter the NSIS installer published on GitHub.
- `anime_data.db` is released separately from `Kunepro/nimlat-dbs`. App and AnimeDB versions are independent.
- The app downloads AnimeDB at runtime. Do not add the released database to the app repository or installer.
- Published tags and binaries are immutable. Fixes use a higher version; never silently replace a published file.
- The `x.y.z` version shape is used because npm, Electron packaging, and the updater require an ordered compatible
  version. Nimlat does not use Semantic Versioning as a product policy: the three fields are monotonically increasing
  release identifiers, and no field promises API compatibility, breaking-change meaning, or a particular release
  scope. The maintainer chooses the next version deliberately; tooling must not infer it from commit messages.

## Hard release gates

Do not publish source code, a tag, or binaries until all applicable items pass.

- [ ] `LICENSE`, `NOTICE.md`, and the in-app About copy consistently allow personal non-commercial self-compilation and
  modification only for contributions submitted through pull requests; redistribution remains prohibited.
- [ ] Project assets, fonts, Electron, and production dependencies have been reviewed for third-party attribution and
  redistribution requirements. Required notices are available from the installed application.
- [ ] The complete Git history and the release worktree have been scanned for secrets, personal data, local databases,
  logs, credentials, certificates, and private keys.
- [ ] The release commit contains no accidental generated files. In particular, review tracked bundles and build output,
  not only ignored `dist/` and `release/` directories.
- [ ] The worktree is clean, all intended fixes are committed, and the app version is not `0.0.0`.
- [ ] `npm run validate` and `npm run audit:prod` pass on the release commit.
- [ ] Any database schema change has explicit written approval, a migration and data-preservation plan, and full
  validation. A release must never silently wipe or rebuild user data.
- [ ] macOS builds are explicitly unsigned, launch after the documented Gatekeeper override, and are described
  accurately
  in the release notes. Never tell users to disable Gatekeeper globally.
- [ ] Windows builds are explicitly unsigned, complete the documented SmartScreen override on a normal consumer device,
  and disclose the `Unknown publisher` warning. Smart App Control or enterprise policy may prevent installation.
- [ ] If this version is submitted to the Microsoft Store, its MSIX/AppX package was built from the same release commit,
  passed Store certification, and is described as a separate Microsoft-signed distribution channel. Do not describe
  the unsigned GitHub NSIS installer as signed.
- [ ] Every installer has been installed and launched on its target operating system.
- [ ] The published AnimeDB release is compatible with this app release and exposes the checksum digest expected by the
  downloader.
- [ ] Release notes are ready and every GitHub release asset reports an uploaded state and GitHub-managed digest.

The macOS configuration intentionally sets `identity` to `null` and disables notarization. A successful macOS package is
therefore expected to trigger Gatekeeper on a clean machine. That warning and the manual override are release behavior,
not a packaging failure.

## Tooling and credentials

Use Node.js 20 or newer and npm 10 or newer. Use `npm ci` in each clean release checkout so every platform builds from
the committed lockfile.

Build native dependencies on the target operating system:

| Target                        | Required build host | Output                            |
|-------------------------------|---------------------|-----------------------------------|
| macOS Apple Silicon and Intel | macOS               | DMG and ZIP for `arm64` and `x64` |
| Windows                       | Windows x64         | NSIS installer for `x64`          |
| Linux                         | Linux x64           | AppImage for `x64`                |

Nimlat bundles `better-sqlite3`, so one host should not be treated as a reliable build environment for all three
operating systems. Never copy `node_modules` between build hosts.

Release tokens belong in protected environment variables or GitHub Actions secrets. Never place them in the repository,
shell history, release notes, or build logs. The release-assembly job uses GitHub's scoped `GITHUB_TOKEN`; local
fallback
builds must not receive a publishing token unless the maintainer explicitly chooses a local upload path.

## One-time authoritative source publication

The GitHub repository is public, so the publication push makes the selected history immediately visible. For the
initial Nimlat publication, the reviewed local `master` is the sole authoritative history. The existing remote history
is disposable bootstrap history and must not be merged back merely because it exists on GitHub.

The final pre-CI publication deliberately uses `--force`, not `--force-with-lease`. At this point `master` is not yet
protected by required CI, and only the copyright owner is authorized to write it; if an unexpected remote commit appears
while protections are absent, replacing it is the intended behavior. This is a one-time bootstrap exception.

After CI and branch protections are enabled, never force-push `master`, with either `--force` or `--force-with-lease`.
Normal changes go through pull requests and required CI. If a pull-request branch legitimately needs rewritten history,
update that branch with `--force-with-lease`; plain `--force` is not permitted after the bootstrap publication.

1. Finish and commit the release fixes. Review tracked, untracked, and ignored files deliberately:

   ```bash
   git status --short --branch
   git ls-files
   git status --short --ignored
   ```

2. Run a full-history secret scanner, not only a search of the current files. If a real credential is found, revoke or
   rotate it before considering any history cleanup. Re-run the scan after remediation.

3. Review exactly what will become public:

   ```bash
   git log --stat --oneline --all
   git diff --check
   git status --short
   ```

4. Prepare the intended local `master` history with meaningful release-source commit messages. Review its complete tree
   and history again; the force push replaces remote-only commits and cannot be used as a substitute for this audit.

5. When the reviewed worktree is clean, connect the local repository if needed and publish the authoritative branch:

   ```bash
   git branch --show-current
   git remote get-url origin
   git push --force -u origin master
   ```

6. Confirm that `git branch --show-current` printed `master` and that the remote commit and tree match the reviewed
   local
   state. On GitHub, set `master` as the default branch and verify
   that `LICENSE`, `NOTICE.md`, `README.md`, and this file render correctly. Enable secret scanning and push protection,
   then review the Security tab before starting the binary release. Commit and run the CI workflow successfully before
   making its status check mandatory; immediately afterwards enable the branch rulesets described below.

Do not create the release tag during the first source push. Create it only after the exact version commit has produced
and passed validation for every platform.

## Branch governance after authoritative publication

### Repository settings baseline

Keep these repository settings aligned with the release and contribution policy:

- Actions may use actions and reusable workflows owned by `Kunepro` plus actions created by GitHub. Third-party
  Marketplace actions are not allowed by default, and full-SHA pin enforcement remains disabled until every referenced
  action has been converted from a release tag to a reviewed commit SHA.
- The default `GITHUB_TOKEN` permission is read-only for repository contents and packages. GitHub Actions cannot create
  or approve pull requests. A workflow or job receives a narrower write permission only where this document explicitly
  requires it, such as `contents: write` for draft-release assembly.
- Workflow runs from every external fork contributor require explicit maintainer approval before consuming a runner.
- Pull requests use squash merge only, with the pull-request title as the resulting commit message. Merge commits,
  rebase merging, and auto-merge are disabled. Successfully merged head branches are deleted automatically, and GitHub
  should suggest updating an out-of-date pull-request branch.
- The personal repository has no collaborators with direct access, no deploy keys, and no webhooks. `Kunepro` is the
  implicit repository owner. Re-audit these lists before each major release or after installing an integration.
- Private vulnerability reporting, the dependency graph, Dependabot alerts and security updates, secret scanning, push
  protection, and secret validity checks are enabled. Dependabot version updates and CodeQL default setup remain
  deferred
  until their behavior and CI cost are reviewed deliberately.

The bootstrap sequence matters:

1. Publish the reviewed authoritative `master`, including `.github/workflows/ci.yml`.
2. Let the `Full validation` job from the `CI` workflow run successfully on GitHub at least once.
3. Configure the required status check only after GitHub knows that check name.
4. Enable the branch rulesets and block direct updates, deletions, and force pushes to `master`.

Keep approval policy and CI enforcement in separate rulesets so their bypass behavior is independent:

- The CI ruleset requires the `Full validation` status check and has no maintainer bypass. No pull request, including
  one
  authored by `Kunepro`, may merge with a missing, pending, or failed required check.
- The review ruleset requires a pull request and the copyright owner's approval for other contributors. `Kunepro` has a
  pull-request-only bypass for the review requirement because GitHub does not allow authors to approve their own pull
  requests. This bypass must not permit direct pushes to `master`.
- `.github/CODEOWNERS` remains the source of review ownership, but it does not replace an enforced ruleset.

The desired result is: contributors can propose code only through pull requests approved by `Kunepro`;
maintainer-authored
pull requests need no self-approval but still require green CI. The one-time pre-CI `git push --force` above must be
completed before these protections are activated and is not part of the normal release procedure. Once protected,
`master` is never force-pushed; only pull-request branches may use `--force-with-lease` when their history must change.

## Per-version release contract and manual fallback

Examples below use `1.0.0`. Replace it with the intended app version everywhere. The AnimeDB version does not determine
the app version.

### 1. Freeze the release source

Start only after feature and fix work is complete:

```bash
npm ci
npm run validate
npm run audit:prod
git diff --check
git status --short --branch
```

Do not package from a dirty worktree. `npm ci` and the validation commands can rebuild native dependencies, so run them
only after other development or validation processes have finished.

### 2. Set and commit the app version

Update both `package.json` and `package-lock.json` through npm:

```bash
npm version 1.0.0 --no-git-tag-version
node -p "require('./package.json').version"
git diff -- package.json package-lock.json
npm run validate:quick
```

Commit the version only after reviewing the diff:

```bash
git add package.json package-lock.json
git commit -m "chore(release): v1.0.0"
git status --short
git rev-parse HEAD
git push origin master
```

Record the commit SHA. Every build host must check out that exact commit. Do not use a different branch tip later and
label its artifacts with the same version.

### 3. Build on native hosts

The release workflow normally performs this step in separate GitHub-hosted jobs. For a manual fallback, use a clean
checkout of the recorded commit on each host and run `npm ci` first.

macOS builds both architectures because they are explicitly configured in `package.json`:

```bash
npm run package:mac
```

Windows x64:

```powershell
npm run package:win -- --x64
```

Linux x64:

```bash
npm run package:linux -- --x64
```

`npm run package` builds only for the current platform. `npm run publish:github` also builds and publishes only from its
current host; it is not a complete three-platform release command. Do not use it as the orchestrator for the first
release.

Each build must report the same app version and source commit. Preserve its complete `release/` output until platform
security behavior and update metadata have been verified.

### 4. Verify platform security behavior and packaged applications

The macOS build is intentionally unsigned and unnotarized. Test the exact DMG on a clean macOS profile:

1. Attempt to open Nimlat normally and confirm Gatekeeper blocks the first launch.
2. Open **System Settings → Privacy & Security**.
3. Select **Open Anyway** for Nimlat and confirm the warning dialog.
4. Verify that subsequent launches work without disabling Gatekeeper globally.

Do not recommend `spctl --master-disable`, removal of the quarantine attribute, or any system-wide security bypass.
Link release users to Apple's current instructions instead.

On Windows, inspect the installer signature in PowerShell:

```powershell
Get-AuthenticodeSignature .\release\Nimlat-1.0.0-x64.exe | Format-List
```

The expected status is `NotSigned`; `package.json` deliberately sets `win.signExecutable` to `false`. Launch the exact
downloaded installer on a normal consumer Windows 11 device and confirm this path:

1. SmartScreen displays **Windows protected your PC** and identifies an unknown publisher.
2. Select **More info**.
3. Select **Run anyway** and complete installation.

Do not tell users to disable SmartScreen or Smart App Control globally. Smart App Control enforcement and enterprise
policy may block unsigned software without a supported per-app override; document those systems as unsupported rather
than weakening their security configuration.

On Linux, mark the AppImage executable and launch the exact file that will be uploaded.

Install or mount the produced artifact rather than launching only an unpacked build. Perform the smoke checklist below
before collecting upload files.

### 5. Collect release assets

The release workflow transfers each native job's output through GitHub Actions artifacts and assembles one draft
release.
For a manual fallback, copy the uploadable files from the three native hosts into one clean `release-upload/` directory.
Upload all applicable electron-updater metadata and blockmaps, not only the visible installers.

Expected inventory:

| Platform | Required assets                                                                                          |
|----------|----------------------------------------------------------------------------------------------------------|
| macOS    | `Nimlat-1.0.0-arm64.dmg`, `Nimlat-1.0.0-x64.dmg`, both ZIP files, applicable blockmaps, `latest-mac.yml` |
| Windows  | `Nimlat-1.0.0-x64.exe`, its blockmap, `latest.yml`                                                       |
| Linux    | `Nimlat-1.0.0-x64.AppImage`, its blockmap, `latest-linux.yml`                                            |

Exact blockmap output may vary by electron-builder version. Treat the generated `latest*.yml` files as the source of
truth: every file named by update metadata must be uploaded under the exact same name. The macOS ZIP target is required
for macOS auto-update metadata even though users will normally download the DMG.

Do not upload unpacked application directories, `builder-debug.yml`, or `builder-effective-config.yaml`.

GitHub calculates and exposes the digest for each uploaded release asset. Nimlat does not generate or upload a separate
checksum file. Confirm the GitHub asset state and digest after upload.

### 6. Run the packaged-app smoke checklist

Run this on macOS arm64, macOS x64 or Rosetta, Windows x64, and Linux x64 as applicable:

- [ ] Installer, DMG, or AppImage opens without an unexpected security failure.
- [ ] Nimlat launches and remains usable at the minimum `800x480` resolution.
- [ ] A clean profile creates its local application data without writing into the repository or installation directory.
- [ ] The published AnimeDB baseline is discovered, downloaded, checksum-verified, opened, and displayed.
- [ ] Existing `user_data.db` and `image_data.db` survive an upgrade test; no wipe or rebuild occurs.
- [ ] Main library navigation, media details, preferences, images, logs, and one representative write flow work.
- [ ] Update check reaches the configured `Kunepro/nimlat` release feed and fails gracefully when no newer release
  exists.
- [ ] Application logs contain no startup, native-module, SQLite, preload, or updater errors.
- [ ] Uninstall behavior has been checked without deleting user-owned data unexpectedly.

The first release has no earlier public version from which to test the complete auto-update path. Test a real
`1.0.0 -> 1.0.1` update before publishing the next release. `electron-updater` requires a signed app on macOS, so Nimlat
hides the integrated app-updater interface there and macOS users update manually from GitHub Releases. The updater IPC
surface intentionally remains available; hiding the unsupported UX is the release policy.

### Optional Windows Microsoft Store channel

The GitHub and Microsoft Store downloads are two distribution channels for the same Windows app release:

- GitHub publishes the unsigned NSIS installer and its electron-updater metadata.
- Microsoft Store receives an MSIX/AppX package built from the same immutable commit and app version. After successful
  certification, Microsoft signs and distributes that Store package at no certificate cost to an individual developer.
- The Store signature applies only to the Store-delivered package. Never copy the signed-status claim to GitHub assets.
- Store submission must not rebuild from a newer branch tip. Preserve or regenerate the submission package from the
  recorded release commit and verify that its bundled application reports the same user-facing version.
- The Store package manifest requires four numeric fields. Map Nimlat `x.y.z` to Store package version `x.y.z.0`; the
  fourth field is reserved for Store use. This mechanical mapping does not make Nimlat a Semantic Versioning product.
- Store certification may finish after the GitHub Release is public. That timing difference does not create a new app
  version as long as the source commit and application version are identical.

The Store package format and Partner Center submission are a separate implementation step. Until that path is tested,
the release workflow should retain the Store package as a CI artifact for manual submission rather than claiming that
publication to Partner Center is automated.

### 7. Tag the validated commit

Only after every platform artifact was built and tested from the recorded SHA:

```bash
git status --short
git tag -a v1.0.0 <RECORDED_COMMIT_SHA> -m "Nimlat v1.0.0"
git show --stat v1.0.0
git push origin v1.0.0
```

Confirm that the tag points to the commit whose `package.json` version is `1.0.0`.

### 8. Assemble a GitHub draft release

The release workflow should perform the following assembly automatically. If it is unavailable, perform the same steps
in `Kunepro/nimlat` manually:

1. Create a draft release from the existing `v1.0.0` tag.
2. Use `Nimlat 1.0.0` as the title.
3. Add release notes using the template below.
4. Upload every collected artifact.
5. Confirm GitHub shows automatic source ZIP and tarball links for the tagged commit.
6. Confirm every uploaded asset reports an uploaded state and GitHub-managed digest.
7. Keep the release as a draft until the final review is complete.

Suggested release notes structure:

```markdown
## Highlights

- <User-visible change>

## Downloads

- macOS Apple Silicon: `Nimlat-1.0.0-arm64.dmg`
- macOS Intel: `Nimlat-1.0.0-x64.dmg`
- Windows x64: `Nimlat-1.0.0-x64.exe`
- Linux x64: `Nimlat-1.0.0-x64.AppImage`

## Data

AnimeDB is downloaded separately by the application from the official `Kunepro/nimlat-dbs` releases.

## Licensing

Nimlat is source-available proprietary software, not open-source software. See `LICENSE` and `NOTICE.md`.

## Platform security

- macOS: Nimlat is unsigned and unnotarized. Follow Apple's **Open Anyway** procedure after the first blocked launch.
- Windows: Nimlat is unsigned and may show **Windows protected your PC**. Select **More info**, verify that the
  installer
  came from the official Nimlat GitHub Release, then select **Run anyway**. Smart App Control or organization policy may
  prevent installation.
```

State both unsigned platform policies prominently. Link Apple's **Open Anyway** instructions, do not imply that the
build
was checked by Apple or Microsoft, and never recommend disabling an operating-system security feature globally.

### 9. Publish and verify

Immediately before selecting **Publish release**:

- [ ] Tag, release title, `package.json` version, filenames, and update metadata all use the same version.
- [ ] No asset is missing, duplicated, or from a different commit.
- [ ] Source availability and third-party notices are correct.
- [ ] Signing status and any known limitations are accurately disclosed.
- [ ] The AnimeDB download smoke test passed without changing the frozen AnimeDB schema.

After publication:

1. Open the public release page in a signed-out browser session.
2. Confirm each public asset exposes the GitHub-managed digest, then download one artifact per platform.
3. Confirm source ZIP and tarball downloads contain `LICENSE`, `NOTICE.md`, and `RELEASING.md`.
4. Install one public download and repeat a short launch smoke test.
5. Confirm supported update checks from an older test build resolve the new release when such a build exists.
6. Monitor GitHub issues and application logs for packaging, update, and database integrity failures.

## Failed release and rollback policy

- Before publication, keep the GitHub Release as a draft and fix/rebuild as needed. If source changes, use a new commit
  and rebuild every affected artifact; do not mix commit SHAs in one release.
- After publication, do not replace assets under the same version. Publish a higher version.
- Never move or reuse a published tag.
- If an installer is unsafe, mark the release clearly and remove it from the recommended download path while preparing
  the patch. Assume downloaded copies still exist.
- If a release threatens user DB or file integrity, stop distribution, explain the affected versions and recovery path,
  and do not introduce a wipe or rebuild without the explicit approval required by the database policy.

## Planned release automation

Release automation is manually initiated by the maintainer through `workflow_dispatch`. It must never run merely because
a pull request was merged or code was pushed to `master`.

The workflow should:

1. Accept or select the exact release commit and intended `x.y.z` app version.
2. Run the full validation and production audit gates.
3. Use separate GitHub-hosted macOS, Windows, and Linux jobs so native dependencies are built on their target operating
   systems.
4. Produce unsigned macOS DMG/ZIP files, an unsigned Windows NSIS installer, and a Linux AppImage with all required
   updater metadata and blockmaps.
5. Optionally produce the Windows Store submission package from the same commit, but keep Store submission manual until
   that channel has been validated end to end.
6. Upload the platform outputs to one GitHub draft release, verify that no commit SHA is mixed in, and publish only
   after
   an explicit maintainer approval.

Only the release-assembly job receives `contents: write`; validation and native build jobs remain read-only. Automation
is complete only when it also verifies version/tag consistency, platform security behavior, updater metadata, artifact
provenance, GitHub asset digests, and the absence of mixed commit SHAs.
